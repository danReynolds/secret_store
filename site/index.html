CLI · macOS · Linux desktop

Secrets,
OS-backed.

Keep secret values protected behind the OS credential store, commit a manifest of non-secret config and kb:// references, then launch any local process with its resolved environment. Five commands; no account, server, daemon, or shell hook.

0.1.0 pre-release. Evaluate the CLI or SDK from a reviewed checkout; published channels are not live yet.

Dart / Flutter SDK

Read secrets directly.

SecretStorage(appId:) applies Keybay's fixed runtime policy across iOS, Android 12+, macOS, and Linux desktop.

SDK quickstart →

Dart

import 'package:keybay/keybay.dart';

final store = SecretStorage(appId: 'com.example.app');
await store.writeString('api-token', tokenFromOAuth);
final stored = await store.readString('api-token');

Storage path by runtime.

The default constructor exposes no backend selector.

ContextStored asProtected by

Apple hardware backing is not attested. Android reports the actual wrapping-key provider. Login-bound stores must be reachable and unlocked.

Windows and headless deployments are unsupported.

Security contract.

Threat model and reporting policy →
No silent downgrade
Locked, missing, invalidated, corrupt, and unsupported stores return typed failures. A plaintext store key is never substituted.
Authenticated, coordinated files
File paths use XChaCha20-Poly1305, key commitment, atomic 0600 writes, fsync, and cross-process locking.
Measured claims only
Keybay reports the resolved scheme. Android reports its wrapping key's observed security level; Apple native items report none.
Explicit boundary
Keybay protects stored values, not a caller after retrieval. Same-user malware, rollback, and root remain outside the threat model.

Local by design.

The CLI stores per-user developer values; the SDK stores values in a local store named by appId. Keybay provides no synchronization, team administration, hosted control plane, or production-secret provisioning.
