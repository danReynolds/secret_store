01 Dart SDK

Keep secrets out of environment variables entirely.

Use SecretStorage directly in Dart and Flutter. One constructor selects the strongest available platform scheme; byte values are the core API.

main.dart 
import 'package:keyway/keyway.dart';

final store = SecretStorage(
  appId: 'com.example.myapp',
);

await store.writeString('api-token', token);
final value = await store.readString('api-token');
  • Pure Dart; no Flutter plugin
  • Bytes-first with string conveniences
  • Typed, value-free failures

The local path

A reviewable contract. A value-free repository.

Project files say which values are needed. Each developer supplies those values to their own operating-system-backed store.

  1. 1

    Commit the contract

    .secrets.env holds public literals and qualified kw:// references. Review sees every environment variable your app will receive.

  2. 2

    Store locally

    Values go through Keychain, Secret Service, Android Keystore, or an authenticated encrypted file whose key lives in the platform keystore.

  3. 3

    Resolve once

    Missing references stop the launch. A complete manifest is injected into exactly one child process; Keyway does not stay in the process tree.

Security posture

Small surface. Loud failures. Measured claims.

Keyway chooses one platform mechanism and reports what it actually resolved. If the required keystore is unavailable, it throws instead of quietly writing plaintext or selecting a weaker home.

Read the design
01

Fail closed

Unavailable, locked, invalidated, corrupt, and migration states remain distinct typed failures. Recovery is explicit.

02

Authenticated at rest

The file scheme uses XChaCha20-Poly1305 with HKDF-SHA256 key commitment, atomic writes, file locking, and fsync.

03

Hardware honestly reported

Apple and Android paths report hardware backing only after platform-specific measurement—not from platform name alone.

04

Real platform receipts

CI exercises macOS Keychain and Linux Secret Service. Repeatable local lanes cover iOS, Android, and entitled macOS paths.

Install honestly

Evaluate the reviewed source today.

The first signed release has not been cut, so pub.dev and Homebrew commands would fail today. Clone the repository now; the release workflow is already designed for pub.dev, signed GitHub binaries, and a dedicated Homebrew tap.

pub.dev after 0.1.0 Homebrew after 0.1.0 GitHub binaries after 0.1.0

Clone, resolve the workspace, then activate the local package:

shell
git clone https://github.com/danReynolds/keyway.git
cd keyway
dart pub get
dart pub global activate --source path packages/keyway_cli
keyway doctor

Clone a reviewed checkout, then point your application at it:

pubspec.yaml
dependencies:
  keyway:
    path: ../keyway

Pre-release consumers should pin the exact reviewed commit. See the release contract.

Context, not a scorecard

Use Keyway when less machinery is the feature.

Keyway is intentionally not a universal secrets platform. These tools solve adjacent jobs well; the right choice depends on whether you need local rigor, provider portability, team administration, or encrypted files.

Tool Best fit Model What you choose
Keyway for Dart Local Dart storage and simple run-scoped injection One automatic OS-backed store per platform Small surface over provider choice, profiles, sync, and team controls
SecretSpec ↗ One declaration across profiles, providers, and languages 11 pluggable local and hosted providers Breadth, generation, auditing, and migration with a larger configuration surface
1Password CLI ↗ Teams already using a centralized 1Password vault op:// references resolved from an account or service identity Sharing, policy, and administration instead of a local-only dependency-free path
flutter_secure_storage ↗ Flutter apps needing broad platforms and biometric or accessibility controls Configurable Flutter plugin with platform-specific options More platform breadth and knobs instead of pure Dart and a standalone CLI
dotenvx ↗ Portable encrypted .env workflows across local, CI, and deploys Encrypted values committed while private keys stay separate Repository portability instead of values that never enter project history

Comparison language is based on each project's current first-party documentation, linked above. Last reviewed July 2026.
